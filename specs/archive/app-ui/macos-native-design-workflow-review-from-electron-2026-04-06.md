# macOS Native Design Workflow Review From Electron (2026-04-06)

Status: Active
Date: 2026-04-06
Owner: xLightsDesigner Team

## Purpose

Use the existing Electron `Design` workflow as a reference artifact while defining the native macOS `Design` screen more precisely.

Electron is a workflow reference only.
It is not the target shell.

## What To Keep

### 1. Creative brief and proposal stay central
Keep:
- creative brief/proposal summary as the first substantive read surface

Reason:
- this page should remain intent-first.

### 2. Rationale and warnings remain secondary
Keep:
- rationale, assumptions, and warnings as supporting context

Reason:
- they help explain the proposal without taking over the page.

## What To Remove

### 1. Live dashboard feel
Remove:
- any operator-dashboard framing
- any sense that this page is monitoring live runtime state

### 2. Technical translation density
Remove:
- sequence translation detail from the main design surface

## What To Change

### 1. Make proposal state clearer
Change:
- the screen should answer whether a coherent proposal exists immediately

### 2. Make open questions readable
Change:
- unresolved creative questions should be visible but compact

## Native Design Screen Definition Derived From Review

1. top summary = creative direction and design status
2. left lower panel = proposal/brief payload
3. right lower panel = rationale, warnings, questions
4. no heavy sequence revision detail

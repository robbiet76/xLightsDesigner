# Sequencer Control Quality Refactor Plan

Status: Active  
Date: 2026-04-27  
Owner: xLightsDesigner Team  

## Purpose

Define the current development plan for improving the sequencer control layer after the first full-sequence benchmarks proved that the app can generate, apply, read back, and score a full-display sequence, but still cannot reliably produce a coherent musical/design composition.

This plan is intentionally not a product phase name. It is a working checklist for the current quality and architecture pass.

## Current Assessment

The sequencing pipeline is no longer primarily blocked by xLights write plumbing.

Working strengths:

- designer context can reach the sequencer through persisted handoffs
- sequence plans can create timing tracks and effect commands
- apply/readback can verify timing, display order, effect creation, clone/copy, delete, update, layer edits, and modal-blocked apply states
- render feedback can score weak output instead of treating readback success as creative success
- read-only production sequence summaries can be used as reference examples without training on or copying source sequences

Current blockers:

- full-display authoring still lacks a first-class composition plan
- the sequencer chooses placements more than it authors visual intent
- Stage 1 effect training is transitional and too small for final-quality sequencing
- `sequence-agent.js` has accumulated too many responsibilities
- render critique is useful but not yet specific enough to drive a strong next pass

## Target Outcome

The sequencer should move from:

1. select effects
2. distribute effects across targets
3. apply and render

to:

1. build an explicit composition plan
2. realize that plan as timing-anchored effects and layers
3. apply in a single full-pass batch when possible
4. score render evidence against the intended composition
5. use critique to revise the composition, not only individual effects
